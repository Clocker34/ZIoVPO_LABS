# Задание 1. Подготовка репозитория (новый серверный репозиторий / перенос с РБПО)

Документ фиксирует **формулировку задания**, **соответствие текущего проекта adboard** и **перечень контроллеров и модулей**, которые нужно сохранять при переносе из старого проекта (РБПО) на GitHub.

---

## Текст задания

1. Создать **Git-репозиторий** для серверной части проекта.
2. Перенести из прошлого проекта в новый репозиторий:
   - **аутентификацию** (JWT access/refresh);
   - **авторизацию** (ролевая модель, правила доступа);
   - **шифрование (HTTPS)**;
   - подключение к БД **PostgreSQL** (или иной используемой реляционной БД).
3. Перенести **переменные и секреты** с репозитория РБПО либо завести новые (не коммитить секреты в открытом виде).
4. Перенести **pipeline** из прошлого проекта в новый, с шагами **test** и **build**.
5. Изучить теорию по **UML**-диаграммам (типы, назначение).
6. Ознакомиться с теорией по **ER**-диаграммам.

Теория UML/ER выполняется **самостоятельно** (лекции, учебник, материалы курса); в код репозитория не входит.

## Соответствие текущего репозитория `adboard`

| Требование задания | Где реализовано в этом проекте |
|--------------------|--------------------------------|
| Git-репозиторий | Корень содержит модуль Maven **`adboard/`**, история коммитов на GitHub. |
| JWT access/refresh | `security/JwtTokenProvider.java`, `service/AuthTokenService.java`, `controller/AuthJwtController.java`, сущность `UserSession`, таблица `user_sessions`. |
| Роли и правила доступа | `entity/User.java` (роль), `config/SecurityConfig.java` (`hasRole("ADMIN")` для `/api/users/**`, `/api/reports/**`), фильтр `JwtAuthenticationFilter`. |
| PostgreSQL | `application.properties`: `spring.datasource.url`, пользователь/пароль через env. |
| Переменные и секреты | Шаблон **`.env.example`** в корне; в приложении подстановки `${JWT_SECRET}`, `${SPRING_DATASOURCE_*}`, bootstrap-админ и т.д. Создайте локальный **`.env`** (не коммитить). |
| HTTPS | В коде Spring Boot приложение по умолчанию **HTTP**. HTTPS обычно включают на **проде**: reverse-proxy (nginx, Traefik), Kubernetes Ingress, или `server.ssl.*` в Spring — это часть **развёртывания**, не отдельного Java-файла. Укажите на защите схему: клиент → TLS → прокси → приложение по HTTP внутри сети. |
| Pipeline test + build | Файл **`.github/workflows/maven.yml`**: отдельные шаги **Test** (`mvn test`) и **Build** (`mvn package`). |

## Контроллеры и связь с аутентификацией

Ниже — **существующие контроллеры** (`adboard/src/main/java/ru/rkjrth/adboard/controller/`). После переноса в новый репозиторий их состав должен сохраняться; защита запросов — через JWT и `SecurityConfig`, а не через отдельный «новый» контроллер на каждый ресурс.

| Файл | Назначение |
|------|------------|
| `AuthJwtController.java` | **POST /auth/login**, **POST /auth/refresh** — выдача и обновление JWT (access/refresh). |
| `AuthController.java` | Регистрация и прочая логика `/api/auth/...` (как в проекте). |
| `UserController.java` | Пользователи; для админ-маршрутов нужна роль ADMIN. |
| `CategoryController.java` | Категории; типичный защищённый GET после JWT. |
| `ListingController.java` | Объявления. |
| `MessageController.java` | Сообщения. |
| `ReportController.java` | Жалобы; админские пути по правилам Security. |
| `CsrfController.java` | Эндпоинт CSRF (актуален, если оставляли cookie/CSRF; в JWT-only API может быть минимально использован). |

**Итог для переноса:** вместе с контроллерами переносятся **фильтр JWT**, **SecurityConfig**, **сервис токенов**, **репозитории**, **сущности** и **настройки БД/JWT**.

## Переменные окружения (что перенести или создать заново)

Минимальный набор для сервера (см. также `.env.example`):

| Переменная | Назначение |
|------------|------------|
| `SPRING_DATASOURCE_URL`, `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD` | Подключение к PostgreSQL |
| `JWT_SECRET` | Секрет подписи JWT (≥ 32 символов для текущей конфигурации) |
| `JWT_ACCESS_MS`, `JWT_REFRESH_MS` | Опционально — TTL токенов |
| `SERVER_PORT` | Порт приложения |
| `ADBOARD_BOOTSTRAP_ADMIN_USERNAME`, `ADBOARD_BOOTSTRAP_ADMIN_PASSWORD` | Первый администратор при пустой БД |

В CI-секретах GitHub (**Settings → Secrets and variables**) храните тестовые значения или используйте только H2 в тестах без реальных паролей продакшена.

## Pipeline

Файл **`.github/workflows/maven.yml`** выполняется на push/PR и содержит два явных шага:

1. **Test** — `./mvnw test` в каталоге `adboard`.
2. **Build** — сборка артефакта (`package`), чтобы подтвердить собираемость после тестов.

При переносе из РБПО замените этот workflow на ваш прежний YAML, если там были дополнительные шаги (линтеры, деплой).

## UML и ER (самостоятельно)

- **UML:** виды диаграмм (классов, последовательности, компонентов, развёртывания и др.), как читать и когда применять — по программе дисциплины.
- **ER:** сущности, связи 1:1 / 1:N / M:N, ключи — для проектирования схемы под PostgreSQL.

К этому репозиторию можно приложить диаграммы отдельными файлам (PNG/PDF/draw.io) по требованию преподавателя.
